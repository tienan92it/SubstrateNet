Architecture

A layered climb up the DIKW pyramid — raw code and conversations at the base, synthesized wisdom at the top — across three databases and one runtime contract. Syntax is deterministic. Meaning is agent-driven. Every claim carries its grounding.

Layered model

The layers climb the DIKW pyramid: Data (L0–L1) → Information (L1.5–L2) → Knowledge (L2.5–L5) → Wisdom (L6). Each layer has its own SQLite table family; edges cross layers explicitly via foreign keys.

Scope × grounding

Two tags on every knowledge node keep objective fact separate from inference. They are the mechanism behind "based on facts, never assume."

scope — technical (architecture) · industry (business domain) · meta.

groundingmeaningsource
structuralobjective, parsed from artifactscode symbols, SQL schema, manifests
statedasserted in a conversationextracted facts
corroboratedstated and matched to a code entitythe Reconciler
externalcited from outside the projectresearch backend (opt-in)
modelthe agent's own inferenceenrichment agents

Project-truth queries default to structural / stated / corroborated. external and model are opt-in and always filterable, so "fill the gap" knowledge never gets mistaken for what your project actually does. subnet status breaks every layer down by both scope and grounding.

Pipeline

Each ingest run pulls new turns, segments them into windows, and routes each window through mechanical cleaning before any classification LLM: session filter → turn normalize → window briefs → pre-triage embed dedupe → triage on briefs → extract with anchor gate → early fact dedupe → batch clusterer → tiered file analyze → fused or deep enrich. See workflow-refactor.md.

Cross-project work runs on top of that, over global.db: the Linker and mechanical matching build L4/L5, then two L6 agents run in order — the KnowledgeOrganizer classifies projects by actionability (PARA), groups skills into proficiency-leveled competency areas, and clusters everything into a subject → topic taxonomy; the WisdomSynthesizer then distills a headline, cross-project insights, and named gaps on top. Both have a data-driven offline fallback so the layer always renders.

Enrichment stages (L2.5)

  • Manifest parser — walks the project (depth-limited) for dependency manifests and infra files; emits dependency / tool facts (structural). Monorepos with nested manifests are fully covered.
  • Domain-from-code — promotes SQL tables to entity facts and foreign keys to relates_to edges (structural).
  • Reconciler — matches stated entities to structural ones by normalized name; links same_as and upgrades grounding to corroborated.
  • Standard profile: domainFuser + industryFuser — two flash-first calls replacing eight frontier enrich agents.
  • Deep profile: legacy stack — TechnicalProfiler, DomainModeler, ArchitectureModeler, IndustryClassifier, IndustryEnricher, domain modelers, etc.
  • Gap detector — evidence-cited knowledge_gap nodes; names the gap, never the answer.

Deterministic stages

  • Window Segmenter — slices a session into windows on user-turn boundaries with size and time caps.
  • Syntax Pass — extracts artifacts that are unambiguous from form: paths, code blocks, shell commands, error messages, stack traces, URLs, ticket IDs.
  • Code Resolver — for any mention (syntax or agent), look it up in code.db and write a k_to_code edge.

Agent stages

  • Triage — drops noise and labels each window (batched; reads window briefs).
  • Pre-triage dedupe — mechanical window drop via embeddings (before triage LLM).
  • Dedupe — embedding similarity for facts; early dedupe before cluster.
  • Extractors — Decision, BusinessLogic, Intent, ProblemSolution. Routed by triage domain so business-logic agents don't waste tokens on chitchat.
  • Clusterer — mechanical attach for high-confidence matches; clustererBatch for the ambiguous band.
  • Summarizer — names and writes a structured summary for each touched concept.
  • Linker — proposes cross-project links the mechanical pass missed.
  • Verifier — periodic sweep for contradictions, supersessions, low-confidence pruning.

The agent layer

A couple dozen chat agents plus one embedding-only agent. Each has a versioned prompt, a JSON schema, and a model reference resolved from config. All share a single runtime that handles caching, validation, and persistence. Agent-bound calls run with bounded concurrency (config.concurrency). The table below is representative, not exhaustive.

Agent runtime

Every agent call goes through one function. Inputs are a typed payload plus the agent's prompt and schema; outputs are validated JSON, cached by content hash, and persisted to agent_runs for audit. 

interface Agent<I, O> {
  name: string;
  promptVersion: number;        // bump invalidates cache
  modelKey?: string;            // overrides config lookup
  schema: JsonSchema;           // strict validation
  prompt(input): ChatMessage[];
  postprocess?(o, input): { output, confidence };
}

interface AgentRuntime {
  run<I, O>(agent, input): Promise<{ output, confidence, model, cached }>;
  // 1. hash(agent.name + model + prompt_version + input) -> cache lookup
  // 2. call backend (Ollama / OpenAI-compatible / Anthropic)
  // 3. parse JSON; validate against schema; one repair retry on failure
  // 4. write to agent_runs (success or failure)
}

Cache invariants

  • Same input + same model + same prompt version → cached hit, zero cost
  • Bump promptVersion → all prior runs ignored, agent re-runs
  • Swap model in config → cache miss only for affected agent; others unchanged

Three databases, three responsibilities. The split keeps the code DB drop-in for codegraph users and lets the knowledge schema evolve independently. Knowledge-schema columns (scope, grounding, …) are added via additive migrations, so existing databases upgrade in place.

Design trade-offs

  • Local-only. No cloud sync. The hard runtime dependency is a local (or pluggable) LLM. The alternative would be regex-based NL understanding, which drifts the moment anyone tries to maintain it.
  • Cached, not deterministic. Agent output is stable per (agent, model, input_hash, prompt_version). Bumping the prompt version invalidates the cache by design.
  • Incremental clustering. The Clusterer attaches greedily and may drift; subnet global link --rebuild triggers a full recompute when the drift becomes visible.
  • Pull-based ingest. No real-time chat hooks by default; run subnet update (or subnet watch / automation) to pick up new sessions. MCP subnet_ingest still works.
  • Code DB stays codegraph-compatible. New code-side fields go in project_metadata JSON rather than schema changes.
  • Conservative triage by default. Windows are kept unless an axis is firmly negative with confidence ≥ 0.6. triage audit --dropped lets you tune.
